# Bank Sync (Open Banking) — Architecture Plan
**Date:** 2026-03-19
**Status:** Future — implement AFTER multi-account model is complete
**Depends on:** `2026-03-19-multi-account-model.md` (requires `bank_accounts` table + `account_id` on transactions)

---

## Why Do This

Manual CSV export/import is the #1 friction point for any personal finance app. Bank sync
eliminates it — transactions appear automatically, in near real-time, without the user
having to remember to export from their bank.

Without bank sync, users must:
1. Log into their bank
2. Navigate to export (often buried 3+ clicks deep, especially in traditional banks)
3. Download a CSV
4. Import it into Trakzi
5. Repeat every week or month

With bank sync, this becomes zero effort after initial setup.

## How It Works

```
Your app                 Aggregator (Nordigen)        User's Bank
────────                 ─────────────────────        ───────────
User taps
"Connect Bank"  ──────►
                         Creates requisition
                         Returns redirect URL
◄──────────────────────
Redirects user to
bank login page  ──────────────────────────────────►
                                                      User authenticates
                                                      Grants consent
                                                      Returns to Trakzi
◄──────────────────────────────────────────────────
Store requisitionId
Fetch account list ────►
                         Returns account IDs
◄──────────────────────
Store in bank_accounts

Nightly / on-demand:
Fetch new transactions ─►
                         Calls bank API
                         Returns transactions
◄──────────────────────
Dedup + insert
(ON CONFLICT DO UPDATE)
```

The aggregator is the licensed Third Party Provider (TPP) under PSD2. Trakzi is just their
API customer — no regulatory burden, no FCA/BaFin registration needed.

## Provider Recommendation

**Primary: Nordigen / GoCardless Bank Account Data**

| Attribute | Detail |
|-----------|--------|
| Coverage | 2,500+ banks across 31 EU countries + UK |
| Pricing | Free up to 50 end-user connections/day; ~€0.50/user beyond |
| Reliability | Good for major EU/UK banks; variable for southern/eastern EU |
| PSD2 compliant | Yes — fully sanctioned by banks, no screen scraping |
| Re-auth cadence | Every 90 days (PSD2 legal requirement, not provider limitation) |
| SDK | REST API, well-documented |

**Fallback: TrueLayer** — better reliability at scale, starts at ~£99/mo.
**US users (if ever): Plaid** — 12,000+ US institutions, $0.30-$0.50/connection/month.

Do not build against multiple providers simultaneously. Start with Nordigen, abstract behind
a `BankSyncProvider` interface so swapping later is a 1-file change.

## Reliability — Honest Assessment

| Region | Uptime | Notes |
|--------|--------|-------|
| UK | ~97% | Open Banking mandate since 2018, well-implemented |
| Germany, France, Netherlands | ~93–95% | Solid PSD2 implementations |
| Spain, Italy | ~85–90% | More variable |
| Eastern EU (some banks) | ~70–80% | Minimum-viable PSD2 compliance |

**Known failure modes — design for all of these:**

1. **90-day consent expiry (law, unavoidable):** User MUST re-authenticate with their bank
   every 90 days. Show expiry warnings 14 days and 3 days before. Never silently fail.

2. **Pending → settled transaction updates:** Banks expose pending transactions that can
   change amount or disappear. Handle via upsert on `external_tx_id`, not insert.

3. **Merchant names are raw garbage:** `"AMZN MKTP DE*1N3X2Y9Q2"` not `"Amazon"`.
   Requires normalization pass (see Phase 4).

4. **Bank maintenance windows:** Common Friday night → Saturday morning for traditional
   banks. Don't alarm users for expected downtime.

5. **Silent connection failures:** Bank changes auth flow → connection returns 401 with no
   warning. Detect via failed sync, mark `sync_status = 'error'`, notify user.

6. **CSV + bank sync deduplication:** If a user imports a CSV AND has sync enabled for the
   same account, every transaction appears twice. The `external_tx_id` unique index (in the
   account model) solves this — but only if the account model is in place first.

## Schema

This plan adds two fields to the `bank_accounts` table defined in the multi-account plan.
They are already included in that schema for reference. No additional tables needed.

Fields used:
```sql
bank_accounts.sync_provider         -- 'nordigen', 'truelayer', 'plaid'
bank_accounts.sync_external_id      -- provider's account ID
bank_accounts.sync_consent_expires  -- PSD2 90-day re-auth deadline
bank_accounts.sync_last_at          -- last successful sync timestamp
bank_accounts.sync_status           -- 'manual' | 'active' | 'expired' | 'error'
```

And on `transactions`:
```sql
transactions.account_id             -- FK to bank_accounts
transactions.external_tx_id         -- provider's transaction ID (dedup key)
transactions.import_source          -- 'csv' | 'bank_sync' | 'manual'
```

The unique index in the account model:
```sql
UNIQUE INDEX idx_transactions_dedup ON transactions(account_id, external_tx_id)
  WHERE external_tx_id IS NOT NULL;
```

This is what prevents duplicates when the same transaction arrives via CSV AND bank sync.

## Implementation

### Phase 1 — Connection flow

**New files:**
- `lib/bank-sync/nordigen.ts` — Nordigen API client
- `app/api/bank-sync/connect/route.ts` — initiate connection
- `app/api/bank-sync/callback/route.ts` — handle return from bank
- `app/api/bank-sync/accounts/route.ts` — list connected accounts
- `app/api/bank-sync/disconnect/route.ts` — remove connection

**Flow:**

  }

```typescript
// app/api/bank-sync/connect/route.ts
export async function POST(req: NextRequest) {
  const userId = await getCurrentUserId()
  const { institution_id } = await req.json()

  const client = new NordigenClient()
  const { link, id } = await client.createRequisition(
    userId,
    institution_id,
    `${process.env.NEXT_PUBLIC_APP_URL}/api/bank-sync/callback`
  )

  // Store pending requisition in Redis (5 min TTL, just for the OAuth roundtrip)
  await redis.setex(`bank_sync:pending:${userId}`, 300, id)

  return NextResponse.json({ redirect_url: link })
}
```

```typescript
// app/api/bank-sync/callback/route.ts
export async function GET(req: NextRequest) {
  const userId = await getCurrentUserId()
  const requisitionId = await redis.get(`bank_sync:pending:${userId}`)

  const client = new NordigenClient()
  const { accounts } = await client.getAccounts(requisitionId)

  for (const externalAccountId of accounts) {
    const details = await client.getAccountDetails(externalAccountId)
    await neonInsert('bank_accounts', {
      user_id: userId,
      name: details.name ?? details.iban,
      account_type: 'checking',  // user can change later
      currency: details.currency ?? 'EUR',
      institution: details.institution_name,
      sync_provider: 'nordigen',
      sync_external_id: externalAccountId,
      sync_consent_expires: new Date(Date.now() + 89 * 24 * 60 * 60 * 1000).toISOString(),
      sync_status: 'active',
    })
  }

  // Trigger initial sync
  await triggerSync(userId)

  redirect('/settings/accounts?connected=true')
}
```

### Phase 2 — Transaction sync

```typescript
// lib/bank-sync/sync.ts
export async function syncUserAccounts(userId: string) {
  const accounts = await neonQuery(
    `SELECT * FROM bank_accounts
     WHERE user_id = $1 AND sync_status = 'active'`,
    [userId]
  )

  for (const account of accounts) {
    await syncAccount(account)
  }
}

async function syncAccount(account: BankAccount) {
  const client = new NordigenClient()

  try {
    const { booked, pending } = await client.getTransactions(
      account.sync_external_id,
      account.sync_last_at?.split('T')[0]  // only fetch since last sync
    )

    for (const tx of [...booked, ...pending]) {
      await neonQuery(
        `INSERT INTO transactions
           (user_id, account_id, external_tx_id, amount, description,
            date, import_source, tx_type)
         VALUES ($1, $2, $3, $4, $5, $6, 'bank_sync', 'expense')
         ON CONFLICT (account_id, external_tx_id)
         DO UPDATE SET
           amount = EXCLUDED.amount,
           description = EXCLUDED.description`,
        [
          account.user_id,
          account.id,
          tx.transactionId,
          Math.abs(parseFloat(tx.transactionAmount.amount)),
          tx.remittanceInformationUnstructured ?? tx.creditorName,
          tx.bookingDate ?? tx.valueDate,
        ]
      )
    }

    await neonQuery(
      `UPDATE bank_accounts SET sync_last_at = NOW(), sync_status = 'active'
       WHERE id = $1`,
      [account.id]
    )
  } catch (error) {
    await neonQuery(
      `UPDATE bank_accounts SET sync_status = 'error' WHERE id = $1`,
      [account.id]
    )
  }
}
```

**Sync triggers:**
- On demand: user taps "Sync now" in Settings → Accounts
- Scheduled: daily cron via Vercel Cron Jobs (`vercel.json` → `crons`)
- Post-import: after any manual CSV import, check if bank sync would overlap

```json
// vercel.json
{
  "crons": [
    {
      "path": "/api/bank-sync/cron",
      "schedule": "0 6 * * *"
    }
  ]
}
```

### Phase 3 — Consent expiry management

PSD2 mandates 90-day re-authentication. Build a proactive warning system:

```typescript
// lib/bank-sync/consent.ts
export async function getExpiringAccounts(userId: string) {
  return neonQuery(
    `SELECT * FROM bank_accounts
     WHERE user_id = $1
       AND sync_status = 'active'
       AND sync_consent_expires < NOW() + INTERVAL '14 days'`,
    [userId]
  )
}
```

UI warnings:
- 14 days before: yellow banner in Settings → Accounts
- 3 days before: orange banner on every page header
- After expiry: red banner, sync paused, button to reconnect

Re-connect flow is identical to initial connect — same API call, new consent token.

### Phase 4 — Merchant name normalization

Raw bank strings: `"AMZN MKTP DE*1N3X2Y9Q2"`, `"SQ *COFFEE SHOP"`, `"PAYPAL *NETFLIX"`

**Option A: Build a normalization table (free, limited)**
```sql
CREATE TABLE merchant_aliases (
  raw_pattern text PRIMARY KEY,  -- regex or prefix
  clean_name  text NOT NULL,
  category    text
);
-- e.g., ('AMZN MKTP%', 'Amazon', 'Shopping')
--       ('NETFLIX%',   'Netflix', 'Subscriptions')
```
Start with the 50 most common merchants for your user base. Grows over time.

**Option B: Ntropy API (paid, ~$0.001/transaction)**
Send raw merchant string, get back: clean name, category, logo URL, merchant website.
Best quality. Add as a post-sync enrichment step.

**Option C: Plaid's Transaction Enrichment add-on**
Only worth it if using Plaid (US users).

**Recommendation:** Start with Option A (own alias table) for the 50 most common EU merchants.
Add Ntropy when merchant quality becomes a user complaint.

## Does YNAB Use It?

Yes — YNAB Direct Import uses Plaid in the US. Their stance is instructive:

- Offered but not promoted as the primary way to use YNAB
- Their docs warn: *"Bank import can create a false sense of security. We recommend
  reviewing every transaction manually."*
- YNAB's philosophy is intentional, manual engagement with money — auto-import can
  undermine the habit they're trying to build

For Trakzi this is less relevant — you're analytics-focused, not building a budgeting
philosophy. Auto-import aligns directly with your UX goal: zero friction data capture.

## Why You Must Do the Account Model First

| Without account model | With account model |
|-----------------------|--------------------|
| Synced transactions land in flat pool | Tagged to specific `bank_accounts` row |
| No dedup key | `external_tx_id` unique per account |
| CSV + sync = instant duplicates | `ON CONFLICT` handles dedup cleanly |
| Credit card double-count worsens | Transfer detection is automatic |
| User adds bank sync, data gets worse | User adds bank sync, data gets better |

**This is the most important architectural constraint in this plan.** Implement bank sync
before the account model and you will make data quality worse, not better.
